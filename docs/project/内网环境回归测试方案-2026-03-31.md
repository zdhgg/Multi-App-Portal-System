# 内网环境回归测试方案

## 1. 背景说明

当前系统部署特征如下：

- 部署环境为内网，不联网
- 操作系统为 Windows 10 64 位
- 系统采用“直接拷贝目录”方式部署
- 目前整体运行较稳定，但偶尔会出现“启动应用失败”

结合项目现状，本次回归测试方案重点不放在互联网依赖，而放在以下真实风险：

- 直接拷贝部署后，`node_modules`、`main-portal/dist`、配置文件或脚本不完整
- Windows 下 PM2 守护进程权限不一致，导致偶发启动失败
- 端口 `8002` 被其他进程占用，导致 PM2 启动后健康检查失败
- `detection-api/.env`、`configs/system-config.json` 等配置文件缺失、编码异常或内容不符合生产要求
- 开机自启、重启恢复、PM2 异常恢复链路在内网机器上不稳定

## 2. 方案目标

本次回归测试目标如下：

- 验证系统在“离线、直接拷贝部署”的前提下可稳定启动
- 验证系统在端口冲突、PM2 权限异常、配置异常等场景下的表现和恢复能力
- 验证系统重启、重装、再次拷贝后，核心功能不回退
- 将“偶发启动失败”问题收敛为可复现、可定位、可验收的问题

## 3. 现状分析与依据

从当前项目脚本和测试来看，系统已经具备一些适合内网回归的基础能力：

- `start-production.ps1` 已内置生产启动前检查，包含 `JWT_SECRET`、端口占用、PM2 进程状态、PID 归属和 `/health` 健康检查
- `Start-Portal.bat` 已包含开始、重启、停止、状态查看、构建前端并重启、开机自启配置等入口
- `check-environment.ps1` 已覆盖 Node.js、npm、PM2、前端构建产物、端口占用、防火墙规则等检查
- 现有自动化测试已覆盖部分高风险点，例如：
  - PM2 进程匹配
  - 应用生命周期启动/停止
  - BOM 配置兼容
  - 配置路径解析

因此，本方案建议采用“自动化检查 + 人工冒烟 + 异常注入 + 恢复验证”的组合方式。

## 4. 回归测试范围

### 4.1 P0 必测范围

这部分每次发布、每次拷贝部署后都必须执行：

- 环境完整性检查
- 一键启动与脚本启动
- 健康检查
- 登录与门户首页加载
- 应用启动、停止、再次启动
- 端口占用处理
- PM2 状态、日志与恢复

### 4.2 P1 重点回归范围

- 开机自启
- 重启机器后的服务恢复
- 配置文件保存、重载和持久化
- Windows 文本编辑器修改配置后的兼容性
- 前端重建后系统可用性

### 4.3 P2 周期性回归范围

- 长时间运行稳定性
- 多次连续启动/停止后的资源释放
- 日志、备份、告警等运维能力

## 5. 测试环境设计

建议至少准备 2 台测试环境，最好 3 台：

- 环境 A：干净安装环境
  说明：模拟新机器，验证“拷贝即部署”是否可靠
- 环境 B：长期运行环境
  说明：模拟当前实际使用环境，验证升级、覆盖拷贝、重复启动是否稳定
- 环境 C：故障注入环境
  说明：专门用于制造端口冲突、PM2 权限异常、配置损坏等问题

每台环境统一记录以下信息：

- 机器名、IP、Windows 版本
- Node.js 版本
- npm 版本
- PM2 版本
- 项目拷贝来源时间
- 是否为管理员启动

## 6. 测试策略

本次回归分为 4 层执行。

### 6.1 第一层：部署完整性检查

目标：先确认离线拷贝包本身可用，避免把“包不完整”误判为程序缺陷。

执行命令：

```powershell
.\check-environment.ps1
```

重点检查：

- `detection-api/node_modules` 是否存在
- `main-portal/dist/index.html` 是否存在
- `ecosystem-prod-loader.config.js`、`start-production.ps1` 是否存在
- 端口 `8002` 是否已被占用
- 防火墙规则是否存在

### 6.2 第二层：启动冒烟回归

目标：验证最关键的启动链路。

执行入口：

```powershell
.\start-production.ps1
```

或双击：

- `Start-Portal.bat`

验证点：

- PM2 是否成功拉起 `portal-api`
- `pm2 pid portal-api` 是否能拿到 PID
- `8002` 端口监听者是否为 `portal-api`
- `http://localhost:8002/health` 返回 `200`
- 门户首页可打开
- 登录成功

### 6.3 第三层：异常注入回归

目标：模拟“偶发启动失败”的高频真实场景。

重点注入场景：

- 端口 8002 被其他进程占用
- PM2 权限异常
- PM2 守护进程异常
- `.env` 缺失或弱密钥
- `main-portal/dist` 缺失
- `node_modules` 缺失或不完整
- 配置文件编码异常或 BOM

### 6.4 第四层：恢复与持久化回归

目标：验证系统在重启、故障、重复操作后的恢复能力。

重点验证：

- `pm2 restart portal-api` 后服务恢复
- `pm2 kill` 后重新执行启动脚本可恢复
- 开启 PM2 自启动后，重启 Windows 后服务自动恢复
- 配置修改、用户数据、日志、备份设置在重启后仍有效

## 7. 详细回归用例清单

| 编号 | 优先级 | 测试项 | 操作步骤 | 预期结果 |
|------|--------|--------|----------|----------|
| RG-01 | P0 | 环境检查 | 执行 `.\check-environment.ps1` | Node/npm/PM2/关键文件检查通过；如有告警必须可解释 |
| RG-02 | P0 | 一键启动 | 双击 `Start-Portal.bat` 选择 `Start` | 服务正常启动，状态页显示 PM2 online、Health 正常 |

| RG-05 | P0 | PM2 重启 | 执行 `pm2 restart portal-api` | 服务可恢复，端口仍归属于 PM2 进程 |

| RG-16 | P1 | BOM 配置兼容 | 使用 Windows 编辑器保存系统配置后重启相关功能 | 配置仍可读取，备份/系统设置不异常 |

| RG-20 | P2 | 长稳运行 | 连续运行 24 小时，期间执行数次访问与启停 | 无异常退出，日志中无持续性严重错误 |

## 8. 建议执行顺序

建议按以下顺序执行，能够最快暴露“启动失败”类问题：

1. 先跑 `RG-01` 到 `RG-05`
2. 再跑 `RG-09`、`RG-11`、`RG-12`、`RG-13`
3. 然后跑 `RG-06`、`RG-07`、`RG-08`
4. 最后跑 `RG-17`、`RG-18`、`RG-19`、`RG-20`

## 9. 执行频率建议

### 9.1 每次代码改动后

执行 P0 冒烟回归：

- `RG-01` 到 `RG-10`

预计耗时：

- 30 分钟到 45 分钟

### 9.2 每周一次

执行完整回归：

- `RG-01` 到 `RG-20`

预计耗时：

- 0.5 到 1 个工作日

### 9.3 每次正式内网发版前

执行发布回归：

- 完整回归
- 至少 1 次重启机器验证
- 至少 1 次覆盖拷贝验证

## 10. 缺陷判定标准

以下情况应直接判定为高优先级缺陷：

- 启动脚本返回成功，但 `portal-api` 实际未监听 `8002`
- `/health` 不通，但界面显示已启动
- 被其他进程占用端口时，系统误判为启动成功
- 同一版本、同一机器上重复执行启动结果不稳定
- 重启机器后服务不自启动或自启动后不可访问
- 配置文件经 Windows 编辑器保存后导致功能失效

## 11. 证据留存要求

每条用例至少留存以下证据中的 2 项：

- 命令行截图
- `pm2 status` 结果
- `/health` 返回截图
- `pm2 logs portal-api --lines 50` 输出
- `detection-api/logs/pm2-error.log` 或 `pm2-out.log`
- 问题出现时间、机器名、操作人

建议固定采集命令如下：

```powershell
pm2 status
pm2 logs portal-api --lines 50 --nostream
Invoke-WebRequest http://localhost:8002/health -UseBasicParsing
netstat -ano | findstr :8002
```

## 12. 准入与准出标准

### 准入条件

- 回归版本已经完成离线拷贝
- 关键配置已准备完成
- 测试机器上 Node.js、npm、PM2 已安装
- 至少有一个可用管理员账号用于权限类场景验证

### 准出条件

- P0 用例全部通过
- P1 用例无阻塞性缺陷
- 不存在“偶发启动失败但无法定位”的问题
- 已产出测试记录、缺陷清单和结论

## 13. 结合当前项目的额外建议

基于项目现状，建议把下面 4 项列为长期固定检查项：

- 每次拷贝部署后先执行 `.\check-environment.ps1`
- 每次发版后固定执行一次 `Start-Portal.bat` 和一次 `.\start-production.ps1`
- 每次发版后固定验证一次端口冲突场景
- 每周固定验证一次 PM2 开机自启和机器重启恢复

如果后续要继续降低“偶发启动失败”，建议再补两个自动化方向：

- 增加一份专门的“启动回归脚本”，自动串联环境检查、启动、健康检查、日志抓取
- 增加针对 `start-production.ps1` 的脚本级自动化测试，覆盖端口占用、PM2 PID 异常、健康检查失败等分支

## 14. 结论

针对当前“内网 + Win10 64 位 + 直接拷贝部署”的运行方式，回归测试不能只做功能点点击，必须把“环境完整性、启动链路、异常注入、恢复能力”作为主线。

本方案建议：

- 以 P0 启动链路回归作为每次发版必测项
- 以 PM2 权限、端口占用、配置异常作为重点异常注入项
- 以开机自启、机器重启、覆盖拷贝作为内网环境特有回归项

按此方案执行后，能够更快定位你现在碰到的“偶发启动应用失败”问题，也能显著降低发版后才暴露启动问题的概率。
