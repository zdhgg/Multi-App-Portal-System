# 日志清理功能使用指南

## 📋 概述

系统日志清理功能用于管理和清理过期的日志缓存记录，防止数据库无限增长，优化系统性能。

## 🔍 问题背景

### 发现的问题

系统设置页面显示日志数量异常高（474,850条），经过分析发现：

| 统计项 | 数据 |
|--------|------|
| **总记录数** | 474,850条 |
| **时间跨度** | 2025-09-27 至 2025-10-29 (约33天) |
| **平均每天** | ~14,389条 |
| **错误日志占比** | 49.6% (异常高) |

### 根本原因

1. **日志缓存表无限增长**
   - `log_cache`表设计用于临时缓存，但变成了永久存储
   - 没有定期清理机制
   - 33天积累了47万条日志

2. **错误日志占比过高**
   - 正常系统error日志应该<5%
   - 可能存在循环错误或频繁失败的操作

3. **数据库性能影响**
   - 数据库文件变大
   - 查询性能下降
   - 磁盘空间占用

## ✨ 解决方案

### 1. 自动清理机制

系统现在会自动清理过期日志：

- **触发时机**：每次日志缓存刷新时检查（每小时一次）
- **保留策略**：默认保留最近7天的日志
- **清理操作**：
  - 删除7天前的日志记录
  - 执行数据库VACUUM优化
  - 记录清理日志

**实现位置**：`detection-api/src/services/logService.ts`

```typescript
private async autoCleanupOldLogs(): Promise<void> {
  // 计算7天前的时间戳
  const cutoffDate = new Date()
  cutoffDate.setDate(cutoffDate.getDate() - 7)
  const cutoffStr = cutoffDate.toISOString().replace('T', ' ').substring(0, 19)
  
  // 删除过期记录
  const deleteStmt = this.db.prepare('DELETE FROM log_cache WHERE timestamp < ?')
  deleteStmt.run(cutoffStr)
  
  // 优化数据库
  this.db.prepare('VACUUM').run()
}
```

### 2. 手动清理功能

#### 前端界面

在**系统设置 > 系统日志**页面，点击"清理日志"按钮：

1. 选择保留天数：
   - 最近1天
   - 最近7天（推荐）
   - 最近30天

2. 查看预计删除数量

3. 确认清理操作

**实现位置**：`main-portal/src/components/settings/SystemLogsPanel.vue`

#### API接口

**POST** `/api/logs/cleanup`

**请求体**：
```json
{
  "retentionDays": 7
}
```

**响应**：
```json
{
  "success": true,
  "data": {
    "deletedCount": 467850,
    "retentionDays": 7,
    "cutoffDate": "2025-10-22 17:44:43",
    "beforeSize": 474850,
    "afterSize": 7000
  }
}
```

### 3. 命令行工具

使用交互式清理脚本：

```bash
node scripts/cleanup-logs.js
```

- 优化数据库

**清理选项**：
1. 保留最近7天的日志（推荐）
2. 保留最近30天的日志
3. 保留最近1天的日志

## 🔧 使用建议

### 日常维护

1. **定期检查日志数量**
   - 在系统设置页面查看日志统计
   - 正常情况下应保持在合理范围（<10,000条）

2. **根据需求调整保留策略**
   - 开发环境：保留1-3天
   - 测试环境：保留7天
   - 生产环境：保留30天

3. **监控错误日志占比**
   - 正常应<5%
   - 如果过高，需要排查根本原因

### 紧急清理

如果日志数量过多导致性能问题：

1. **使用命令行工具快速清理**：
   ```bash
   node scripts/cleanup-logs.js
   ```
   选择选项1（保留最近7天）

2. **或通过API清理**：
   ```bash
   curl -X POST http://localhost:3000/api/logs/cleanup \
     -H "Content-Type: application/json" \
     -d '{"retentionDays": 7}'
   ```

3. **验证清理效果**：
   - 刷新系统设置页面
   - 查看日志数量是否降低
   - 测试系统响应速度

## ⚠️ 注意事项

### 数据安全

1. **清理操作不可逆**
   - 删除的日志无法恢复
   - 清理前确认保留策略

2. **重要日志导出**
   - 清理前可先导出重要日志
   - 使用"导出"功能保存为JSON/CSV/TXT

3. **生产环境谨慎操作**
   - 建议在低峰期执行
   - 先在测试环境验证

### 性能影响

1. **VACUUM操作**
   - 会锁定数据库
   - 大数据库可能需要几分钟
   - 建议在低峰期执行

2. **清理时机**
   - 自动清理在后台执行
   - 不影响正常业务
   - 手动清理可能短暂影响日志查询

## 🐛 故障排查

### 清理失败

**问题**：清理操作失败

**可能原因**：
- 数据库被锁定
- 磁盘空间不足
- 权限问题

**解决方案**：
1. 检查数据库连接
2. 确认磁盘空间充足
3. 检查文件权限
4. 查看错误日志

### 日志数量仍然很高

**问题**：清理后日志数量仍然很高

**可能原因**：
- 日志产生速度过快
- 错误日志过多
- 保留天数设置过长

**解决方案**：
1. 排查错误日志来源
2. 修复导致错误的问题
3. 调整保留天数
4. 优化日志级别配置

### 自动清理不工作

**问题**：自动清理机制未生效

**可能原因**：
- 日志服务未启动
- 缓存刷新未触发
- 代码未部署

**解决方案**：
1. 重启detection-api服务
2. 检查日志服务状态
3. 验证代码版本
4. 手动触发清理
